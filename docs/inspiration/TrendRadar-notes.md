# TrendRadar 借鉴笔记

> [TrendRadar](https://github.com/sansan0/TrendRadar) 是 46k stars 的开源舆情监控项目。
> 它和我们的 scout-agent 在采集 + 关键词过滤 + 推送这条主线上高度重合，但定位与目标用户完全不同。
> 这份笔记记录我们**借鉴什么、放弃什么、为什么**，避免开发过程中"看见好功能就想加"。

## 它和我们的根本差异

| 维度 | TrendRadar | llmx-scout-agent |
|------|-----------|------------------|
| 用户数量 | 几万人，各种使用场景 | 1 个人（先），最多几十个创作者（未来） |
| 输出形态 | 推送消息 + HTML 网页 | **结构化源文件** |
| 下游 | 用户的眼睛 | 另一个 agent 程序 |
| 关注点 | 信息覆盖广、不漏热点 | 信息深度足、judgment 空间大 |
| 时效性 | 越快越好 | 当天准点即可，质量 > 速度 |

差异决定了：**它是"通用舆情雷达"，我们是"特定布道者的素材准备员"**。

## 借鉴清单（直接拿来用 / 改造后用）

### 1. 关键词配置语法（直接借鉴）

TrendRadar 的 `frequency_words.txt` 语法很优雅，直接复用：

```
# AI 基础
AI
LLM
/\bagent\b/ => AI Agent
+RAG          # 必须词
!训练营       # 过滤词（标题含此词直接排除）

# 框架与工具
LangChain
LangGraph
+多智能体
```

实现要点：
- 大小写不敏感
- 支持 `/pattern/` 正则
- 支持 `=> 备注` 起别名（推送/日志显示用）
- `+` 必须词、`!` 过滤词
- `[组别名]` 分组（每组独立的 `+/!` 上下文）

### 2. URL 标准化（借鉴）

TrendRadar 解决了"微博 URL 带动态参数导致重复推送"的问题。我们的去重逻辑也需要：
- 移除常见 tracking 参数（utm_*、ref、source 等）
- 统一域名 www 前缀
- HN 项目用 `item?id=` 标准化（去掉评论楼层 anchor）

### 3. 增量去重的两层机制（借鉴）

- 短期：URL hash + ETag/Last-Modified（同一文章不重复抓取）
- 长期：标题相似度（同一事件不同 URL 不重复打包，比如多家媒体报同一新闻）

### 4. 数据源分层重试（借鉴思路）

TrendRadar 对数据源的失败容忍很好——某个平台挂了不影响其他平台。我们也要：
- 每个 source 独立 try/except
- 失败记到日志和 metrics，不阻塞流水线
- 连续失败 N 次告警（写到日志或可选通知，**不要**阻塞下次跑）

## 放弃清单（明确不做）

### 1. 100+ AI provider via LiteLLM
我们只用 Anthropic Claude。多 provider 是开源项目讨好用户的选择，对我们是无谓复杂度。

### 2. 8 个推送渠道
通知降级为可选层。即使做，1-2 个就够（钉钉 / 邮件 / macOS 本地通知 三选一二）。

### 3. AI 翻译模块
我们的内容是中文产出，不需要把英文源翻成中文喂给下游——下游 LLM 直接吃英文原文。
（如果未来要做面向中国用户的"原文摘要中文版"附加在 pack 里，那是后续 v1.1 的事。）

### 4. MCP 服务
TrendRadar 提供 MCP 让用户用 Claude Desktop 查询历史数据，是给"信息消费"场景的。
我们的 pack 文件就是输出，下游 agent 直接读文件，不需要 MCP 中间层。

### 5. Web 配置编辑器
我们的"用户"就一个人，YAML/TXT 直接编辑器改完事。
未来如果要做也是"pack 浏览器"，不是配置编辑器。

### 6. Docker / GitHub Actions / S3 三套部署
本地 cron + SQLite 一套搞定。
（部署到服务器也是直接 systemd timer，不上 Docker。）

### 7. 复杂调度系统（timeline.yaml + 5 种预设）
crontab 写两三行就够。"每天 9 点和 21 点跑一次"不需要 yaml dsl。

### 8. 独立展示区、AI 分析推送、HTML 报告等"展示型"功能
我们的展示是 source pack 文件本身，不需要额外的 HTML 报告页面。

### 9. RSS 订阅源功能
v0.1 不做。HN/GH/Reddit 三个核心源够了。
未来想加 RSS 也可以，但要警惕——RSS 源很容易让平台爆炸式增加，违背"少而精"。

## 可以偷的具体技术点

如果某些环节卡住，可以去 TrendRadar 仓库直接借鉴：

- **关键词解析器**：`trendradar/` 包里的关键词匹配实现
- **URL 标准化**：v4.0.3 引入的标准化逻辑
- **HN/Reddit JSON 端点的处理细节**：直接看它怎么调用的
- **分批推送处理**：如果将来要做钉钉通知且消息超长，它有现成方案

但**不要**直接 fork 或 import 它的包——
- 它的代码是为多场景设计的，复杂度对我们是负担
- LICENSE 是 GPL-3.0，import 会传染
- 我们的目标是 < 2000 行干净代码

## 一个判断准则

每次想加一个新功能时，问自己：
> "这个功能是让 source pack 质量更高，还是只是让 scout 自己更花哨？"

只有前者值得做。
